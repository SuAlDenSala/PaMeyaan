Architecture Overview

The Etoda Gateway acts as the central brain. It handles public user authentication and securely routes authenticated user data to specialized internal microservices.

  • API Protocol: RESTful (FastAPI)
  • Primary Endpoint: /api/*
  • Database: MongoDB (Motor AsyncIO)
  • Authentication: Stateless JWT (Bearer)
2

Local Environment Setup

To run the Gateway locally, ensure Python 3.10+ is installed. Create a .env file in the root directory based on app/core/config.py:

.env
# --- Server Configuration ---
PROJECT_NAME="eTODA Bongao API"
PORT=2011

# --- MongoDB Database ---
MONGODB_URL=mongodb://localhost:27017
DATABASE_NAME=etoda_db

# --- Security & JWT ---
SECRET_KEY=super-secret-key-for-tcto-lgu
ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=120

Authentication Headers

When communicating with protected routes, mobile and web frontends must attach their active JSON Web Token (JWT). The Gateway uses the get_current_admin and commuter dependencies to validate sessions statelessly.

Authorization: Bearer <your_jwt_token_here>

Core REST Operations

Authentication & Security

POST /api/auth/login

Unified OAuth2 login for LGU Admins and Public Commuters. Returns JWT Bearer token.

Identity & Users

POST /api/commuters/register

Public commuter registration. Non-regular statuses pend LGU approval.

POST /api/drivers/login

Authenticates a Driver using LGU-issued cryptographic QR Code hash.

Offline-First Synchronization

GET /api/sync/pull

Mobile client requests updates. Returns drivers, fares, and community alerts.

POST /api/sync/push

Uploads queued offline incident reports into central DB.

Transport Data

GET /api/fares/

Fetch LGU Fare Matrix (Origin, Destination, Specific Fares).

POST /api/incidents/

Queue and record local transit incident reports tied to driver QR.
